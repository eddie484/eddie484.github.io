컴파일러 개발 도중, 테스트를 위해 컴파일러가 만들어낸 코드를 cpu 위에서 돌려보니 결과가 엉망진창이었다.
이전에 Instruction Memory/Data Memory를 reg 배열 메모리에서 IP로 변경하는 과정에서 온갖 타이밍 문제가 발생했던 것이 기억나, IP 미사용 버전에서 테스트해봤더니 정상적인 결과가 나왔다. 파형을 확인해보니 store 직후 load 하는 경우 문제가 발생하고 있었다.
IP 적용 과정에서는 컴파일러 개발 전이었어서 직접 작성한 어셈블리들로 테스트를 돌려본 후 문제가 발견되지 않아 통과시켰었는데, 테스트케이스들을 작성할 때는 고려하지 못했었던 상황이었던 것이다. 하지만 거의 모든 연산이 결과를 메모리에 저장하고, 이후 다시 불러와 다음 연산을 진행한다는 현재 컴파일러 구조 상, store 다음 명령어로 load가 바로 오는 경우가 매우 빈번해 이 문제를 해결하지 못한다면 cpu의 동작 자체가 불가능한 문제이다.

문제의 원인을 확인하니 역시 동기 메모리인 RAM IP에서 데이터 출력에 시간이 필요하다는 점이었다. 기존의 방식에서는 한 사이클 늦게 나오는 메모리 출력을, 출력에 연결된 파이프라인 레지스터 없이 와이어로 연결해 사이클을 맞췄다, 이 방식의 경우 store 이후 다른 명령어가 사용된다면 사이클 딜레이의 영향을 받지 않고 출력을 이용할 수 있었다. 하지만 실제 메모리 내부에서는 한 사이클 늦게 저장이 이루어지고 있으므로, 다음 명령어를 수행하는 중에는 저장이 완료되지 않은 상태이다. 따라서 다음 명령어로 load가 오는 경우 잘못된 값을 읽어왔던 것이다.

이러한 문제를 해결하기 위해 가장 먼저 떠올랐던 방식은, store 이후 load 명령이 오는 경우에는 사이에 한 사이클 stall을 걸어 저장이 완료된 이후 load가 진행되게 하는 것이었다. 하지만 거의 대부분의 연산에서 store-load 세트가 발생하는 현재 컴파일러의 특성 상, 이 방법은 과도한 딜레이의 증가로 성능 손실이 너무 커 채택하지 않았다.
다음으로 떠올린 방식은, 메모리의 출력을 바로 fetch/memory level의 출력으로 사용하는 것이었다. 이렇게 하니 출력값은 정상적으로 나오지만, 해당 단계 내부에서 사용하는 메모리 출력 값이 타이밍이 맞지 않아, branch predictor나 m_fwd 같은 신호의 값에 문제가 생긴다.
마지막으로 구상한 방식은, 모든 문제가 메모리의 저장이 한 사이클 늦게 되는 것이라는 점에서 떠올린 해결책으로, 기존의 타이밍 문제를 해결하던 방식인 메모리 출력을 wire로 하던 것을 역으로 이용하는 방식이다. 메모리 출력을 레지스터를 거치지 않고 사용하는 것이 아닌, 메모리의 입력을 레지스터를 거치지 않고 넣어주는 방법이다. 메모리에 직접 연결될 와이어를 pipereg_F와 pipereg_M에 만들어, instr을 읽어올 pc와, dmem에 들어가는 신호들은 레지스터를 우회해 이전 단의 값을 바로 가져온다. 이를 통해 pc = 100인 순간 imem은 pc = 104를 다루고 있고, store 명령어 A가 execute를 수행하는 사이클에 dmem 역시 명령어 A를 처리하게 되므로, Fetch가 끝나는 시점, Memory가 끝나는 시점에서 RAM IP도 데이터 처리를 완료하게 되므로, IP와 CPU 사이의 사이클 불일치 문제가 해결되는 것이다.
